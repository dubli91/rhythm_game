**Learnings / decisions (Milestone 6):**

- Loop timeline (`src/features/practice/schedule.ts`): a session is chained cycles (1-bar count-in + pattern) with T[k+1] = T[k] + beats┬╖60/bpm[k] тАФ drift-free by construction (10-loop closed-form equality pinned in tests). Beats are BPM-independent (cycle k = session beats [k┬╖beats,(k+1)┬╖beats)) so BPM changes never move rendered notes; only judge/click TIMES change. BPM changes apply at the next cycle (locked LOOP_PREP_AHEAD_SEC=0.25s before the boundary).
- One judge per cycle on a shared session-ms axis; inputs route to the previous cycle's judge until LOOP_HANDOFF_MS=300ms past its end тАФ provably unambiguous for BPM 60тАУ400 and snap тЙд1/32 (window overhang тЙд231.25ms < 300 < 350ms тЙд next window start; proof pinned as a test). Judge note times are offset-free session ms; the SongClock applies the global offset to input times тАФ same convention as song play, so offset semantics can't diverge.
- Metronome: `src/lib/audio/sfx.ts` (synthClickBuffer sine├Чexp-decay + createSfxScheduler, audio-playback.md MUST 8) scheduled at exact ctx times through effectsBus; strong 1720Hz on bar starts, weak 1080Hz elsewhere, clicks run through the pattern bars too (decision: spec only demands the count-in; a continuous metronome substitutes for the absent music). Clicks already in the past are skipped on catch-up after a rAF stall.
- Renderer reuse via `hud: 'practice'` init option in render.ts: gauge widgets simply aren't created (PlayFrameView.gauge is now nullable), an infoText multi-line stats block renders at RENDER_LAYOUT.INFO_X/Y instead. Practice controller rebuilds the info string only when a value changes.

**Learnings / decisions (Milestone 7, settings):**

- `src/features/settings/model.ts` is the headless interaction model (navigation cursor, capture mode, value mutations); screen.ts is presentation-only over it. The model mutates the SAME live `settings` object the shell passes to sessions тАФ one source of truth is what makes "change offset then play immediately" work with no refresh (MUST 11). settings.v1 is written ONLY by the settings screen (write-through per change); the shell only reads + normalizes at boot (clampGlobalOffsetMs, clampVolume, isValidKeyMap fallback to defaults).
- keyMapLanes invariant: always a valid map (8 unique non-empty codes). Duplicate capture rejected with conflict lane highlighted; per-lane reset REFUSED when the lane's default code is currently bound to another lane (would silently invalidate the map); reset-all mutates the array in place (shell holds the reference).
- `RESERVED_LANE_CODES` is exported from input.ts derived from CONTROL_CODES, so settings-screen MUST 10 and the in-play control keys can never drift.
- Calibration is an in-settings MODAL, not a screen-machine state (resolves the spec's open question тАФ app-shell-navigation.md MUST 1 has no calibration screen in the enum). It uses an offset-free SongClock; proposal = mean raw tap error because play-time ╬┤ = raw╬┤ тИТ globalOffset, so offset := mean(raw╬┤) centres the distribution. Taps predating the first click by more than half a period are ignored тАФ e2e must wait past the 0.5s click lead-in before tapping (this bit the first e2e run).
- Sliders are built once and render() only updates values/classes тАФ rebuilding an <input type=range> mid-drag kills the drag. Focused native widgets own keys per app-shell-navigation.md MUST 17 (Escape blurs); key capture is a model-level mode, not DOM focus, and eats every key first.
- Enter/any non-reserved code IS bindable to a lane (spec only reserves the six in-play codes); menu handlers don't consult the key map so this is safe.
- The screen keydown listener is attached on activate()/detached on deactivate() (practice-editor pattern); the shell's global keydown ignores SETTINGS entirely. DOM listeners added during dispatch of an event don't fire for that same event, so the KeyO that opens the screen can't leak into it.

## Key risks (with mitigations)

- **Cross-browser timing precision:** performance.nowтЖФAudioContext.currentTime drift causes judgement error тЖТ `getOutputTimestamp()` with calibrated sampling fallback; assert ┬▒2ms median ╬┤ stability over 10 replays (audio-playback.md).
- **60fps budget on 2,000-note charts:** per-frame allocation/GC stalls тЖТ sprite pooling, culling, no per-frame allocs; measure p95 frame-time тЙд16.7ms in 10-min autoplay (playfield-rendering.md).
- **Backgrounded-tab rAF pauses** decoupling notes from audio тЖТ derive currentBeat solely from the audio clock every frame; test tab-return alignment (playfield-rendering.md).
- **IndexedDB quota/eviction** losing practice patterns тЖТ atomic writes, quota-error surfacing (practice-mode.md, `src/lib/storage/idb.ts`).
- **Autoplay-policy audio lock** before first gesture тЖТ create/resume AudioContext only after first key/click (title screen); handle suspended state (audio-playback.md, app-shell-navigation.md).
- **Logic/render entanglement blocking headless tests** тЖТ judgement/gauge/scoring pure and event-driven in `src/lib` + features; renderer strictly read-only (judgement-scoring.md, playfield-rendering.md).
