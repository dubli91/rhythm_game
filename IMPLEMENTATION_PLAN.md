**Learnings / decisions (Milestone 6):**

- Loop timeline (`src/features/practice/schedule.ts`): a session is chained cycles (1-bar count-in + pattern) with T[k+1] = T[k] + beats·60/bpm[k] — drift-free by construction (10-loop closed-form equality pinned in tests). Beats are BPM-independent (cycle k = session beats [k·beats,(k+1)·beats)) so BPM changes never move rendered notes; only judge/click TIMES change. BPM changes apply at the next cycle (locked LOOP_PREP_AHEAD_SEC=0.25s before the boundary).
- One judge per cycle on a shared session-ms axis; inputs route to the previous cycle's judge until LOOP_HANDOFF_MS=300ms past its end — provably unambiguous for BPM 60–400 and snap ≤1/32 (window overhang ≤231.25ms < 300 < 350ms ≤ next window start; proof pinned as a test). Judge note times are offset-free session ms; the SongClock applies the global offset to input times — same convention as song play, so offset semantics can't diverge.
- Metronome: `src/lib/audio/sfx.ts` (synthClickBuffer sine×exp-decay + createSfxScheduler, audio-playback.md MUST 8) scheduled at exact ctx times through effectsBus; strong 1720Hz on bar starts, weak 1080Hz elsewhere, clicks run through the pattern bars too (decision: spec only demands the count-in; a continuous metronome substitutes for the absent music). Clicks already in the past are skipped on catch-up after a rAF stall.
- Renderer reuse via `hud: 'practice'` init option in render.ts: gauge widgets simply aren't created (PlayFrameView.gauge is now nullable), an infoText multi-line stats block renders at RENDER_LAYOUT.INFO_X/Y instead. Practice controller rebuilds the info string only when a value changes.
- input.ts got an `extraControlCodes` option (base CONTROL_CODES always win conflicts) + `bpmUp`/`bpmDown` PlayControlActions (repeat-on-hold); practice maps Equal/Minus (PRACTICE_CONTROL_CODES). This is the extensibility seam noted in M5.
- Session size: targetLoops (editor LOOPS field, 0 = until-Escape) else floor(20000/notesPerLoop), capped at 200 loops — the renderer receives the whole session as one synthetic chart up front (loop-major note order; sessionIndex = loop·N + patternIndex).
- Practice writes NOTHING to records — controller.ts has no records import at all; e2e asserts records.v1 is byte-identical across a practice session. Accuracy in stats = EX% (comparable with song results); combo does not chain across loops (per-loop scorer, bestMaxCombo is best single loop).

Buttons blur after click so Space/Enter stay grid-owned.
- Patterns persist in the IndexedDB `practicePatterns` store (keyPath patternId) via `src/features/practice/store.ts`; corrupt rows are dropped on read by the isPracticePattern guard (list keeps valid ones).

**Learnings / decisions (Milestone 7, settings):**

- `src/features/settings/model.ts` is the headless interaction model (navigation cursor, capture mode, value mutations); screen.ts is presentation-only over it. The model mutates the SAME live `settings` object the shell passes to sessions — one source of truth is what makes "change offset then play immediately" work with no refresh (MUST 11). settings.v1 is written ONLY by the settings screen (write-through per change); the shell only reads + normalizes at boot (clampGlobalOffsetMs, clampVolume, isValidKeyMap fallback to defaults).
- keyMapLanes invariant: always a valid map (8 unique non-empty codes). Duplicate capture rejected with conflict lane highlighted; per-lane reset REFUSED when the lane's default code is currently bound to another lane (would silently invalidate the map); reset-all mutates the array in place (shell holds the reference).
- `RESERVED_LANE_CODES` is exported from input.ts derived from CONTROL_CODES, so settings-screen MUST 10 and the in-play control keys can never drift.
- Calibration is an in-settings MODAL, not a screen-machine state (resolves the spec's open question — app-shell-navigation.md MUST 1 has no calibration screen in the enum). It uses an offset-free SongClock; proposal = mean raw tap error because play-time δ = rawδ − globalOffset, so offset := mean(rawδ) centres the distribution. Taps predating the first click by more than half a period are ignored — e2e must wait past the 0.5s click lead-in before tapping (this bit the first e2e run).
- Sliders are built once and render() only updates values/classes — rebuilding an <input type=range> mid-drag kills the drag. Focused native widgets own keys per app-shell-navigation.md MUST 17 (Escape blurs); key capture is a model-level mode, not DOM focus, and eats every key first.
- Enter/any non-reserved code IS bindable to a lane (spec only reserves the six in-play codes); menu handlers don't consult the key map so this is safe.
- The screen keydown listener is attached on activate()/detached on deactivate() (practice-editor pattern); the shell's global keydown ignores SETTINGS entirely. DOM listeners added during dispatch of an event don't fire for that same event, so the KeyO that opens the screen can't leak into it.

**Learnings / decisions (Milestone 7, CN):**

- CN semantics (judgement.ts): the HEAD is the note's single scored judgement (CN = 1 note, maxEX = notes×2 — resolves the "CN note counting" ambiguity). A head hit parks the note in 'held' and opens a per-lane hold; release inside the end window (endTimeMs − good, inclusive w/ epsilon) → cnComplete (no scoring/gauge effect, silent on the HUD); earlier release → cnBreak ("treated as BAD": combo reset + gauge BAD delta + BP + kills FULL COMBO, but NOT added to grade counts — same extra-event precedent as emptyPoor). advance() auto-completes a still-held hold at endTimeMs, so holding through the end is never punished and the late window edge is unreachable by construction; a missed head never opens a hold (one note, one missPoor). Autoplay is press-only and relies on the auto-complete.
- JudgementKind gained 'cnBreak'/'cnComplete'; ScoreSummary gained cnBreakCount; gauge tables map cnBreak → BAD delta (HARD <30% halving applies naturally), cnComplete → 0. Shell/records/practice-stats needed no changes (they consume aggregated ScoreSummary fields only).
- validate.ts now enforces the CN span rule the judge relies on: on a lane, any note at beat <= an earlier CN's endBeat is rejected (INCLUSIVE — a note exactly at the tail would need an instant release+repress). scripts/builtin-songs/lib.mjs assertChartInvariants mirrors the same rule so generation fails loudly.
- Renderer (render.ts): CN body = pooled stretched sprite under the head chip (tail→head span, lane color at alpha 0.42 pending / 0.9 held / 0.2 broken-or-missed); while held the head chip pins to the judgement line and the body drains; a broken CN scrolls off dimmed with no chip. Cull cursor treats a CN as gone only when its TAIL passes the bottom edge.
- Autoplay stall bug found+fixed in controller.ts: after a rAF stall the catch-up loop can press two same-lane CNs in one frame, and the second onInput would replace the first still-active hold in the judge's activeHolds map, leaving it stuck 'held' (chip pinned at the line forever). Fix: drain judge.advance(noteTime) before each autoplay press (safe: all earlier notes already pressed, later notes outside the bad window).
- CN showcase content (builtin-song-content.md SHOULD 15): Neon Cascade only — the beats-60-64 riser charts as a held CN whose tail lands exactly ON the STOP beat (pins the same-beat rule for tails; builtinCharts.test.ts asserts it), and the outro's sustained 2-beat bass notes chart as CNs (NORMAL 194 notes / 5 CN, HYPER 333 / 9 CN). Zero audio changes — both sounds were already sustained, so the WAV is byte-identical; only chart JSONs + index.json changed. First Light untouched. verify-e2e.mjs Neon smoke extended to 30s (through the riser CN + STOP) asserting clean counts (BAD0/POOR0/BP0).
- Practice mode is deliberately CN-free (practice-mode.md MUST 4: tap-only patterns) — its controller ignores keyup for judging and doesn't map cnBreak/cnComplete to render states. Structurally unreachable today (pattern schema has no endBeat); if CN ever comes to practice, port the play controller's onRelease wiring + NOTE_HELD/NOTE_BROKEN mapping.

**Learnings / decisions (Milestone 7, song preview):**

- `src/features/select/preview.ts` is a headless preview player (injectable ctx/fetch/timers; 14 unit tests in preview.test.ts). Decided behavior for the one-line SHOULD: the preview follows the cursor's SONG (moving across a song's chart rows is a no-op), starts PREVIEW_DEBOUNCE_MS=300ms after the cursor settles (no churn while scrolling), fades in/out over PREVIEW_FADE_MS=250ms (song switches crossfade), loops the catalog excerpt seamlessly via AudioBufferSourceNode loopStart/loopEnd, and routes through musicBus (MUSIC volume tier governs it). `previewLoopBounds` clamps bad metadata (start beyond buffer → 0; invalid end → buffer end) so an empty/out-of-range loop is unrepresentable.
- Audio is fetched+decoded lazily on FIRST hover and cached per songId for the session — index.json stays the only boot-time fetch (song-library.md MUST 2 intent). A load-token guard prevents a slow fetch from starting audio for a song the cursor already left. Fetch/decode failure console.warns (NOT console.error — both e2e scripts fail on console.error), returns to idle, and clears the target so re-hovering retries; a broken preview can never crash select.
- Shell wiring is exactly two hooks calling one idempotent `updatePreview()`: the tail of renderSelectList() (every selection-changing path re-renders) and a second machine.onChange listener (leaving SONG_SELECT silences the preview; returning resumes it — settings/practice/results round-trips auto-resume from the decode cache). createScreenMachine supports multiple onChange listeners (handlers are a Set). Preview state mirrors into `data-preview-state`/`data-preview-song` attributes on the select screen root — verify-e2e.mjs gained a preview step asserting playing→follows-cursor→idle-in-settings→cache-resume.
- The catalog `preview: {startMs, durationMs}` field (builtin-song-content.md MUST 7) finally has its consumer; entries without a catalogEntry (imported, vestigial) simply never preview. Caveat: WAV audio means one preview hover fetches the full uncompressed song (~MBs) — fine at 3 songs, revisit if the catalog grows (the M2-learnings ogg-encoder item would also shrink this).

**Learnings / decisions (Milestone 7, menu SFX):**

- Menu SFX (audio-playback.md MUST 8) landed 2026-07-17 — this was the LAST unimplemented MUST across all specs; everything still open is SHOULD-tier. `createMenuSfx(ctx, destination)` in src/lib/audio/sfx.ts pre-synthesizes three buffers via the new `synthToneSequenceBuffer` (multi-segment sine, same 3ms-attack + exp-decay envelope per segment; `synthClickBuffer` now delegates to it as the single-segment case — byte-identical output, pinned by a delegation-invariant test). Palette: move = single 1568Hz tick (35ms, amp .35); confirm = rising 988→1319Hz; cancel = falling 659→494Hz (two-tone 50+90ms, amp .5) — musical pitches so rapid overlapping cues stay consonant.
- Wiring: the kit is created in shell.ts `unlockAudio()` on the effects bus (EFFECTS volume tier governs it, MUST 9); `playMenuSfx()` is null-safe (menus are unreachable before the title-gesture unlock, so pre-unlock = silence, never an error). - Settings screen takes an optional `playMenuSfx` hook (move/confirm only); value adjustments (←/→) stay deliberately silent everywhere — volume rows already give live audible feedback (the effects preview click) and a tick on top would double-sound them. Capture mode and the calibration modal eat keys before the hook, so no cues fire during those.
- 465 unit tests green (8 new in sfx.test.ts: tone-sequence length/boundary/peak/delegation + menu kit reuse/distinctness/immediacy/cancelAll); both e2e scripts pass unchanged — menu cues fire throughout verify-e2e.mjs's full navigation and produce zero page errors.

## Key risks (with mitigations)

- **Cross-browser timing precision:** performance.now↔AudioContext.currentTime drift causes judgement error → `getOutputTimestamp()` with calibrated sampling fallback; assert ±2ms median δ stability over 10 replays (audio-playback.md).
- **60fps budget on 2,000-note charts:** per-frame allocation/GC stalls → sprite pooling, culling, no per-frame allocs; measure p95 frame-time ≤16.7ms in 10-min autoplay (playfield-rendering.md).
- **Backgrounded-tab rAF pauses** decoupling notes from audio → derive currentBeat solely from the audio clock every frame; test tab-return alignment (playfield-rendering.md).
- **IndexedDB quota/eviction** losing practice patterns → atomic writes, quota-error surfacing (practice-mode.md, `src/lib/storage/idb.ts`).
- **Autoplay-policy audio lock** before first gesture → create/resume AudioContext only after first key/click (title screen); handle suspended state (audio-playback.md, app-shell-navigation.md).
- **Logic/render entanglement blocking headless tests** → judgement/gauge/scoring pure and event-driven in `src/lib` + features; renderer strictly read-only (judgement-scoring.md, playfield-rendering.md).
